### Profile结论

`embedding_f16x8_kernel`：这个kernel使用了int tx = threadIdx.x * 8;然后手动展开的方式去加载数据，虽然逻辑上是可以合并访存的，但是事实上设备每次会用8条load/store标量指令去加载数据，在ncu报告里会体现
在sectors(扇区)非常多，上百万了。

`embedding_f16x8_pack_kernel`：这个kernel就是向量化读取的，ncu报告里的sectors就少就几千个，而且此kernel速度快约2倍了。

### 需要记住的东西

1. sector，扇区：32 Bytes对齐的最小单位，这个和内存里的扇区有点像。
2. L1 cache line: 128 Bytes的大小
3. 要注意，视角要是warp级别，例如一个线程一次是向量化读取/写入是128 bits=16B，一个warp就是32*16B的数据量了。
4. 内存事务: 32字节或者128字节(某些硬件开启了L1缓存后)。当一个线程束（Warp，包含 32 个线程）同时执行一句读取或写入全局内存的代码时，GPU 的内存控制器不会傻傻地发出 32 次独立的内存请求，而是会把这 32 个线程的访问需求“打包”。这个“打包并由硬件实际执行的单次数据传输过程”，就是一次内存事务。
5. 合并访问与非合并访问举例
- 完美情况：合并访问（Coalesced）假设 Warp 中的 32 个线程，恰好顺序读取了内存中连续的 32 个 float 类型数据（每个 float 占 4 字节，总共 $32 \times 4 = 128$ 字节），并且首地址是对齐的。线程需求：128 字节。硬件执行：GPU 只需要发出 4 次 32 字节的内存事务（或者 1 次 128 字节的事务），就能完美满足这 32 个线程。吞吐量：100%。没有浪费任何带宽。
- 糟糕情况：非合并访问（Uncoalesced）假设同样的 32 个线程，因为索引计算错误（例如跳跃访问或随机访问），访问的空间支离破碎，每个线程访问的地址都隔得很远。线程需求：依然是 128 字节。硬件执行：因为地址不连续，硬件被迫为每个线程（或每几个线程）单独发起内存事务。极端情况下，可能需要发起 32 次 32 字节的内存事务（总共传输了 $32 \times 32 = 1024$ 字节）。吞吐量：$\frac{128}{1024} = 12.5\%$。高达 87.5% 的带宽被浪费了，程序会变得极慢。
